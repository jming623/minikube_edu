# Kubernetes 핵심 구성요소

---

## 전체 구조

```
┌─────────────────── Control Plane (마스터) ───────────────────┐
│  kube-apiserver ←→ etcd                                     │
│  kube-scheduler                                             │
│  kube-controller-manager                                    │
└─────────────────────────────────────────────────────────────┘
        ↕ API 통신
┌─────────────── Worker Node (각 노드) ───────────────────────┐
│  kubelet  ←→  Container Runtime (containerd 등)             │
│  kube-proxy                                                 │
└─────────────────────────────────────────────────────────────┘
```

---

## Control Plane

| 구성요소 | 한 줄 역할 |
|----------|-----------|
| **kube-apiserver** | 모든 요청의 관문. kubectl·kubelet·컨트롤러 전부 여기로 통신 |
| **etcd** | 클러스터 상태(리소스 정보)를 저장하는 분산 저장소 |
| **kube-scheduler** | 새 Pod를 어느 노드에 올릴지 결정 |
| **kube-controller-manager** | Deployment→ReplicaSet→Pod 수 유지 등 상태 조정 루프 |

---

## Worker Node

| 구성요소 | 한 줄 역할 |
|----------|-----------|
| **kubelet** | 노드 에이전트. API 서버가 지시한 Pod를 실제로 기동·감시·보고 |
| **kube-proxy** | Service IP → Pod IP 트래픽 전달 (iptables/IPVS) |
| **Container Runtime** | 이미지 pull, 컨테이너 생성·삭제 (containerd, CRI-O 등) |

---

## 클라이언트/도구 (상주 프로세스 아님)

| 이름 | 역할 |
|------|------|
| **kubectl** | 사용자 CLI. apiserver에 요청을 보내는 클라이언트 |
| **kubeadm** | 클러스터 초기 생성(init/join)용 부트스트랩 도구. 설치 후에는 상주 안 함 |

---

## 동작 흐름 (예: kubectl apply)

```
1. kubectl apply → kube-apiserver → etcd에 의도(desired state) 저장
2. controller-manager: "Pod 2개 필요한데 0개" → ReplicaSet에 Pod 생성 요청
3. scheduler: 새 Pod에 적절한 노드 배정
4. 해당 노드 kubelet: 컨테이너 런타임으로 Pod 기동
5. kube-proxy: Service IP로 들어온 트래픽을 해당 Pod로 전달
```

---

## Minikube에서는 어떻게 대응되나

일반 K8s 클러스터는 Control Plane·Worker가 물리적으로 분리되지만,
Minikube는 **Docker 컨테이너 하나가 "가상 노드" 한 대** 역할을 한다.

```
┌── minikube 컨테이너 (노드 1개 = Docker 컨테이너 1개) ──┐
│  Control Plane 전부 + Worker 전부가 함께 동작           │
│  (apiserver, etcd, scheduler, controller,              │
│   kubelet, kube-proxy, containerd)                     │
└────────────────────────────────────────────────────────┘
```

### 비교표

| 항목 | 실제 K8s (운영) | Minikube |
|------|----------------|----------|
| 노드 | 물리/VM 여러 대 | Docker 컨테이너가 노드 역할 |
| Control Plane | 전용 마스터 노드 | 첫 번째 노드에 같이 들어감 |
| 멀티노드 | 별도 머신 join | `--nodes=N` 또는 `minikube node add` |
| 네트워크 | 클라우드 LB, 실제 IP | 호스트와 격리 → `minikube service` 터널 필요 |
| kubeadm | 직접 실행 가능 | minikube가 내부적으로 대신 처리 |
| Container Runtime | 자유 선택 | 기본 containerd (Docker 드라이버와 별개) |

### `minikube start --nodes=2` 일 때

```
Docker 컨테이너 1 (minikube)       ← Control Plane + Worker
Docker 컨테이너 2 (minikube-m02)   ← Worker만
```

- 첫 번째 노드가 마스터 겸 워커
- 추가 노드(`minikube node add`)는 워커 전용
- 각 컨테이너 안에 kubelet·kube-proxy·containerd가 돌고 있음
